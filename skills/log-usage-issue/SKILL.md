# log-usage-issue

記錄使用問題：$ARGUMENTS

## 步驟 1：理解問題情境

從 `$ARGUMENTS` 取得問題描述，若資訊不足則詢問：

1. 「觸發這個問題的操作或指令是什麼？」
2. 「工具預期應該做什麼，實際上做了什麼？」

若描述已足夠詳細，跳過詢問。

## 步驟 2：根因分類

依以下判斷樹分類：

```
問題是否符合工具文件中描述的功能範圍？
├── 否 → 工具限制（工具設計上不支援此場景）
└── 是 → 行為是否符合文件描述？
         ├── 否 → 工具 bug（行為與文件不符）
         └── 是 → 使用方式錯誤（操作步驟或環境設定有誤）
```

典型情境：

| 根因分類 | 典型情境 |
|---------|---------|
| 使用方式錯誤 | 呼叫 skill 時缺少必要參數；執行順序錯誤 |
| 工具限制 | 僅支援特定框架；目錄結構超出預期；monorepo 不支援 |
| 工具 bug | 相同輸入每次結果不同；產生無效程式碼；工具崩潰 |

## 步驟 3：blocking 判斷

**判定為 blocking（符合任一）：**
- 無法繼續完成當前任務
- 工具產生錯誤輸出且影響專案程式碼
- 根因為「工具限制」且在目前工作流中無法繞過

**判定為非 blocking：**
- 有替代方法可完成相同目標
- 問題僅影響特定邊緣案例
- 根因為「使用方式錯誤」且已知正確用法

輸出判斷結果與理由，讓開發者確認或修正。

## 步驟 4：寫入日誌

讀取 `tool-use-logs/tool-adoption-log.md`，取得下一個 ISSUE-NNN 編號。
若檔案或目錄不存在，先建立。

寫入格式：

```markdown
### ISSUE-NNN

- **問題描述**：<描述>
- **發生時機**：使用階段
- **是否 blocking**：是 | 否
- **根因分類**：<分類>
- **處理方式**：未處理
- **記錄時間**：<YYYY-MM-DD HH:MM>
- **狀態**：開放中
```

## 步驟 5：blocking 後續處理

**若 blocking：**

1. 告知開發者此問題已標記為 blocking（ISSUE-NNN）
2. 詢問：「是否現在進行客製化以解決此問題？」
3. 開發者同意 → 呼叫 `/add-customization ISSUE-NNN`
4. 開發者拒絕 → 更新該記錄：
   - 處理方式：待人工決策
   - 狀態：待人工決策
   - 告知「已標記為待人工決策，停止自動處理。」

**若非 blocking：**
- 告知「已記錄為 ISSUE-NNN，繼續工作。」

## 注意事項

- 根因分類與 blocking 判斷均向開發者透明呈現，允許修正
- 不在未詢問開發者的情況下自動呼叫 add-customization
